# GitHub Strategy

## 目的

本文档定义 `Project-Agent-OS` 在 GitHub 上的稳定呈现方式，避免仓库在公开演进过程中出现定位漂移、文档层级混乱，或示例内容反向主导方法论的问题。

这份文档主要回答四个问题：

1. `README` 应该负责什么
2. `docs/` 应该负责什么
3. `examples/` 应该怎样摆放
4. `roadmap / issues / milestones` 应该怎样分层协同

## 仓库定位

`Project-Agent-OS` 是一个仓库级 operating methodology。

它的核心目标不是展示某一次 AI 项目成果，而是沉淀一种可复用的长期协作方法：

`Agent + Workspace + Memory + SOP + Thread Handoff`

因此，GitHub 上的所有结构都应服务于一个原则：

先讲清方法论，再展示示例，而不是反过来。

## 1. README 负责什么

`README.md` 是 GitHub 访问者的第一入口，只负责让第一次进入仓库的人快速理解：

1. 这个仓库是什么
2. 它解决什么问题
3. 它由哪些核心层组成
4. 从哪里开始读
5. 示例项目在仓库中的角色是什么

### README 应坚持的边界

README 应该：

1. 保持短而清楚
2. 优先给出仓库身份和核心公式
3. 只提供必要导航，不承载全部细节
4. 明确 `examples/` 是应用示例，不是仓库本体

README 不应该：

1. 堆太多执行细节
2. 变成完整手册
3. 被某个垂直领域示例主导
4. 把临时实验写成稳定规则

### README 的稳定结构建议

README 长期应以以下结构为主：

1. 仓库定位
2. Core Formula
3. 解决的问题
4. 系统层级
5. 文档入口
6. Quick Start / First Run
7. 示例说明

## 2. docs/ 负责什么

`docs/` 是仓库的主方法论层，负责承载 README 不应展开的稳定内容。

### docs 的职责

`docs/` 应负责：

1. 核心术语定义
2. 系统架构说明
3. 各模块规则
4. GitHub 演进规则
5. 线程协作与写回原则

### docs 的分层原则

当前建议继续保持以下分层：

1. `docs/system/`
   放仓库级方法、术语、系统总览、GitHub 策略
2. `docs/memory/`
   放 memory 结构与写回规则
3. `docs/sop/`
   放 SOP 的创建与使用规则
4. `docs/handoff/`
   放 thread handoff 规则与模板

### docs 的稳定要求

如果某个结论会影响未来线程如何理解仓库，应优先写入 `docs/`，不要只留在聊天里。

## 3. examples/ 应该怎样摆放

`examples/` 的作用是证明方法可以落地，而不是重新定义方法。

### examples 的职责

示例项目可以用来展示：

1. 一个真实或近真实的项目目录结构
2. memory 文件如何写
3. SOP 如何组织
4. thread summaries 如何保存
5. 方法论如何在某个方向上落地

### examples 的边界

示例项目不应该：

1. 被误认为仓库核心定义
2. 替代 `docs/` 中的方法论说明
3. 夹带品牌化约束来反向污染主仓库
4. 把临时素材堆成仓库主叙事

### examples 的稳定展示方式

对 `examples/` 的推荐策略是：

1. 在 README 中只做简短说明和导航
2. 在示例目录自己的 `README.md` 中讲清应用场景
3. 明确标注“这是示例，不是唯一标准形态”
4. 当多个示例并存时，按应用方向分开，而不是混在主方法层里

## 4. roadmap / issues / milestones 怎样分层

GitHub 的演进管理需要把“长期方向”和“可执行事项”分开。

### roadmap 负责什么

`roadmap` 负责表达仓库未来要长成什么样，属于方向层。

它适合记录：

1. 方法论还缺哪些核心模块
2. 文档结构要怎样完善
3. examples 未来扩展到哪些方向
4. GitHub 对外呈现怎样逐步成熟

### issues 负责什么

`issues` 负责表达具体可执行工作，属于任务层。

它适合记录：

1. 某个文档缺失
2. 某个模板需要补齐
3. 某个 SOP 定义不清
4. 某个示例需要规范化

### milestones 负责什么

`milestones` 负责把多个 issue 组织成阶段性交付，属于节奏层。

它适合表达：

1. `v0.1` 初版公开可读
2. `v0.2` 文档结构稳定
3. `v0.3` 示例层扩充
4. `v1.0` 形成可复用仓库方法框架

## 推荐协同关系

推荐采用以下关系：

```text
Roadmap
  -> 说明阶段目标和结构方向
Issues
  -> 拆成具体待办
Milestones
  -> 把同阶段 issue 组织为一次版本推进
```

## 发布与更新原则

### 主线程 A 的职责

主线程 A 负责：

1. 决定仓库身份
2. 审核 README 主叙事
3. 审核 docs 分层
4. 决定哪些内容属于核心方法
5. 决定是否公开发布某次结构更新

### 辅助线程 B 的职责

辅助线程适合承担：

1. 示例素材采集
2. 样本资产整理
3. issue 草案撰写
4. roadmap 候选项整理
5. 非核心文档草稿补充

辅助线程不应独立决定仓库定位，也不应绕过主线程直接重构核心叙事。

## 当前建议的下一步

按当前仓库状态，GitHub 层最自然的后续动作是：

1. 增加一个公开的 roadmap 文档或 GitHub Project 说明
2. 设计首批 issues 分类
3. 逐步把现有英文说明补成更适合公开传播的中英结合版本
4. 决定这个元仓库是否也要自用一套 `knowledge-base/` 与 `thread-summaries/`

## 决策规则

如果一个改动会影响 GitHub 访问者对仓库身份的理解，就先按“核心方法论变更”处理，而不是当作普通文档小修。
