# Repository Roadmap

## 目的

本文档定义 `Project-Agent-OS` 作为公开 GitHub 仓库的阶段性发展方向。

它回答的不是“今天做什么小修改”，而是“这个仓库下一阶段应该长成什么样”。

## 当前阶段判断

当前仓库已经具备：

1. 基础 README
2. 系统总览
3. 术语层
4. Memory / SOP / Handoff 基础文档
5. 模板
6. 一个 AI 音乐示例项目
7. GitHub 呈现策略

当前更准确的阶段应定义为：

`v0.1 Public Foundation`

即：

仓库已经可以公开访问和理解，但仍处在“方法论骨架已成，公开协作结构待补齐”的阶段。

## 版本路线

### v0.1 Public Foundation

目标：

让 GitHub 访问者能快速理解仓库定位，并看到最小可读的方法论骨架。

完成标准：

1. README 可读
2. 核心 docs 可导航
3. 示例与方法论边界清楚
4. 仓库已公开

状态：

已完成主体建设。

### v0.2 Structure Stabilization

目标：

让仓库从“能看懂”升级为“能协作、能扩展、能持续维护”。

重点工作：

1. 补充公开 roadmap 和 issue 分类
2. 决定元仓库自用的 memory 结构
3. 稳定 docs 层级与 cross-link
4. 强化中英结合的公开表述

完成标准：

1. 有明确 roadmap
2. 有稳定 issue taxonomy
3. 主仓库自身开始实践方法论
4. 新线程能靠本地文档继续推进

### v0.3 Example Expansion

目标：

让仓库展示“同一套方法可用于多个方向”，而不被单一示例绑定。

重点工作：

1. 规范现有示例目录
2. 视需要增加第二个非 AI 音乐示例
3. 统一示例 README 结构
4. 明确示例与核心方法的边界规则

完成标准：

1. 至少有一套规范示例结构
2. examples 不反向主导仓库定位
3. 新示例接入方式清楚

### v1.0 Reusable Operating Methodology

目标：

把 `Project-Agent-OS` 稳定为一套真正可迁移、可复用、可协作的仓库级 operating methodology。

完成标准：

1. 核心术语稳定
2. GitHub 结构稳定
3. 文档分层稳定
4. 模板可复用
5. 至少一到两个高质量示例
6. 多线程协作规则清楚
7. 元仓库自身已使用这套方法运行

## 当前优先级

当前最重要的不是扩张内容量，而是补齐“公开协作结构”。

优先级建议如下：

1. 先稳定 roadmap / issues / milestones 的管理方式
2. 再稳定元仓库自用的 memory 与 handoff
3. 再逐步优化公开表达与双语呈现
4. 最后再扩大 examples 层

## 暂不优先事项

当前不建议优先推进：

1. 一次性扩写过多领域示例
2. 把 README 写成长手册
3. 在方法论未稳定前引入过多自动化脚本
4. 过早承诺完整产品化路线

## 下一阶段建议动作

从当前状态进入下一阶段，最自然的动作是：

1. 建立 issue 分类文档
2. 建立元仓库自己的 `knowledge-base/`
3. 建立元仓库自己的 `thread-summaries/`
4. 更新 README 以反映中英结合定位

## 判断规则

如果一个新增内容不能明显增强“可复用性、可协作性、可续接性”三者之一，就不应优先进入主仓库。
